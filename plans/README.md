  of the `VoxelForest`/`CityGrid` refactor. (003 tests `github.ts`, not the

  is the recommended one.

## Findings considered and rejected

These were surfaced during the audit and deliberately NOT turned into plans, so
they are not re-audited next run:

- **"`Math.max(...)` on an empty array → `-Infinity`/`NaN`"** (VisualizationShell.tsx:56,60):
  rejected — both call sites are guarded (`if (pastDays.length === 0) return []`
  precedes line 56; line 60 is guarded by `fullTerrain.length > 0`). The real
  zero-data edge case is a *different* problem, covered by plan 006.
- **"Stale closure in `handlePlayToggle`"** (VisualizationShell.tsx:176): rejected —
  the keydown effect lists `handlePlayToggle` in its dependency array (line 264),
  so it re-registers when the callback's deps change. Not stale.
- **"Off-by-one in the progressive-reveal `count` loop"** (VoxelForest.tsx:146):
  rejected — correct for cells sorted by `col`; all cells with `col < revealedCols`
  are contiguous at the front of the array, so `count = i + 1` is right.
- **"`HoverInfo` SSR hydration mismatch from `window.innerWidth`"**: rejected —
  the component only renders client-side (inside `is3D && hoveredDay`, under an
  `ssr:false` dynamic import); the SSR fallback path is never taken on a hover.
- **"Rapid play-toggle / year-change accumulates intervals"** (TimelineRuler.tsx:79,
  VisualizationShell.tsx:120): rejected — both effects clear the prior interval
  before creating a new one and on unmount; React runs effect cleanup before the
  next effect.
- **Security review: clean.** Token is server-only (`process.env.GITHUB_TOKEN`,
  no `NEXT_PUBLIC`), username is a typed GraphQL variable (no injection),
  `error.tsx` leaks no internals. (`pnpm audit` not run — no `node_modules`.)
  One non-finding to keep in mind: `/[username]` is an unauthenticated proxy to
  GitHub using the app's token, so a public deploy could have its rate limit
  burned by enumeration. Only act on this if the app is deployed publicly.
